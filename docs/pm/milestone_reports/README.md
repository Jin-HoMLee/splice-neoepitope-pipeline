# Milestone closure reports

Self-contained HTML closure reports, one per milestone, produced **before** a
milestone closes. One artifact serves three layered audiences — **PM
retrospective · lab seminar · portfolio showcase**. Generated by
[`scripts/pm/milestone_report.py`](../../../scripts/pm/milestone_report.py);
design spec at
[`docs/superpowers/specs/2026-06-16-milestone-closure-report-design.md`](../../superpowers/specs/2026-06-16-milestone-closure-report-design.md)
([Issue #752](https://github.com/Jin-HoMLee/splice-neoepitope-pipeline/issues/752)).

## Files per milestone

```
docs/pm/milestone_reports/<slug>.html          # generated artifact (committed)
docs/pm/milestone_reports/<slug>.narrative.md   # author-owned sidecar (committed)
```

`<slug>` = the milestone name lowercased with non-alphanumeric runs collapsed to
hyphens (e.g. `pm-i6 - PM Tooling, Memory & Methodology II` →
`pm-i6-pm-tooling-memory-methodology-ii`).

**Section ownership.** Sections 1/2/6 (header, at-a-glance metrics, inventory
appendix) are script-generated from board #9 + `gh` data on every run. Sections
3/4/5 (Deliverables, Carried-forward & routing, Retrospective) are author-owned
and live in the `.narrative.md` sidecar. The script **never overwrites** an
existing sidecar — on re-run it regenerates the HTML from the (possibly
hand-edited) sidecar plus fresh board data. The first run seeds the sidecar from
a compact digest of the lead role's lab-notebook entries dated within the
milestone window.

## Authoring flow (author-editor-critic)

A closure-ritual **convention, not a hard gate** — closing milestones is
infrequent, so a machine gate would be over-machinery.

| Triad role | Mapping | Step |
|---|---|---|
| **Author** | the milestone's **lead role** (Scientist / Developer) | writes/refines the Deliverables narrative in the sidecar |
| **Editor** | **PM** | edits, writes Carried-forward & routing + Retrospective, requests review |
| **Critic** | **`@-claude review`** on the PR (existing infra) | flags flaws |

1. **Scaffold** — at the milestone-health beat, when a milestone is ready to
   close, PM runs `milestone_report.py <milestone>`. Emits the HTML with data
   sections filled + seeds the narrative sidecar.
2. **Author** — the lead role writes the Deliverables narrative on the branch.
3. **Edit** — PM edits, adds Carried-forward/routing + Retrospective, requests
   the bot review (critic).
4. **Finalize** — address review → merge PR → close the milestone (the close is
   still the PM-coordinated routing act; the report is its evidence). Link the
   merged report from the milestone-close comment and the PM lab-notebook entry.

## Running

The script lazy-imports `jinja2` + `markdown`, so run it under a Python that has
them (the pyenv default does; or `pip install -r scripts/pm/requirements.txt`).
**Not** the `snakemake` conda env — `markdown` is absent there and we keep that
env pristine.

```bash
# full render (seeds sidecar on first run, then renders HTML)
python scripts/pm/milestone_report.py "pm-i6 - PM Tooling, Memory & Methodology II"

# metrics only, write nothing
python scripts/pm/milestone_report.py "<milestone full name>" --dry-run
```

## Metrics

Headline only, computed as pure (unit-tested) functions —
[`tools/ci/test_milestone_report.py`](../../../tools/ci/test_milestone_report.py):
duration (days), throughput (closed/week), avg & median cycle time (days),
per-role closed counts. Cumulative-flow / burn-down charts and status-transition
timing are out of scope (YAGNI; the board has no native transition timestamps).
